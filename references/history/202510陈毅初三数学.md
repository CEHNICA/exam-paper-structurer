# 202510陈毅初三数学 —— 处理详情

## 概况
- 来源：4张手机拍照（微信图片_2025-10-10_000922_951/000930_343/000935_109/000939_836.jpg），每张是一次两页合影，共8个物理页面
- 内容：某中学教研共同体 2025—2026学年度第1学期9月八年级数学月月清（命题教师姓名已脱敏，不适合公开仓库）
- 题量：25题（10选择+5填空+10解答），初二几何（平行四边形/矩形/菱形单元），题干里几乎每题都配图
- 这是这个skill的第一次实践，当时还没有 `match_images_by_position.py`，配图靠拼联系表凭形状肉眼比对

## 关键发现（问题 → 原因 → 解决）
1. **第13题和第17题的插图配反了**
   - 现象：两题都是"四边形+对角线交于O点"结构的图，人工对着一整页的联系表挑图时看走眼
   - 原因：肉眼比对形状这个方法本身对"结构相似但细节不同"的图形容错率低；而且当时唯一的复检手段也是重新看一遍联系表，等于用同一个方法验证同一个方法，没有引入独立信号
   - 解决：用户指出后手动交换了两个文件。之后反思意识到 `content_list.json` 本身就是按阅读顺序排列的，写了 `match_images_by_position.py` 按"图片夹在哪两个题号之间"来配，重新跑这份卷子验证，两题精确得到正确结果（MD5核对一致）——这是该方法后来成为默认第一步的直接原因
2. **MinerU 的 OCR 对手写批注/背面透字很不可靠**
   - 现象：手写圈的答案字母、纸张背面透光的"鬼影"文字经常被读错或读丢
   - 解决：题干等印刷体内容改成直接读原始高分辨率照片，不依赖OCR文本
3. **MiniMax MCP 工具报 `invalid api key`**
   - 原因：账号是中国大陆 Token Plan，MCP 走的是国际站 `api.minimax.io`，认证不过
   - 解决：改用直连 `https://api.minimaxi.com/v1/text/chatcompletion_v2`，图片理解模型用 `MiniMax-M3`（不是 `MiniMax-VL-01`）
4. **（后续被判定为不必要的工作）识别手写答案**：这次实践里花了不少功夫用 Claude+MiniMax-M3 交叉核对选择题/填空题的手写答案。用户之后指出这不是任务目标（见凤城高一数学那份详情），skill 已改成默认不做这一步——这份卷子的 `questions.json` 里仍保留着当时提取的 answer 字段（用户明确说保留不动，不用回头删）

## 数据
- 单张照片裁剪图数量：15~30张不等，一份8页卷子总共约70+张裁剪图
- 真正对应题目插图的：约24张（Q1、Q16两题无配图）
- 噪声占比：约2/3是学生红笔/铅笔解题过程或纸张背面透字的"鬼影"，被 MinerU 误判成图片

## 操作记录
mineru_extract.py 解析 → 拼联系表+肉眼比对配图（后来证明不可靠）→ Claude读图整理JSON → MiniMax-M3交叉核对手写答案（现已改为默认不做）→ latexify_questions.py 转LaTeX（一次性完成，无异常）→ render_markdown.py

## 一句话总结
插图配题号别凭形状肉眼比对，容易把结构相似的图配反；应该一开始就用位置匹配。
