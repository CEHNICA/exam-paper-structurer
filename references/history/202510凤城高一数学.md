# 202510凤城高一数学 —— 处理详情

## 概况
- 来源：1份PDF（微信图片_20251028070013_223_61.jpg.pdf），QuarkXPress生成，pdfinfo/PyMuPDF核实实际2页（每页是两页合影照片，约4个物理页面）
- 内容：2025级高一上学期质量检测一 数学试题（集合/命题/不等式/函数）
- 题量：19题（8单选+3多选+3填空+5解答），代数/函数为主，仅2题有印刷插图（第9题4张函数图象子图+第17题1张销售量分段函数图）
- 这次是 `match_images_by_position.py` 建成后第一次实战应用（陈毅初三数学那次是事后补跑验证，这次是从头就用它）

## 关键发现（问题 → 原因 → 解决）
1. **系统提示的PDF页数不可信**
   - 现象：环境提示这份PDF有"20页"，实际用 `pdfinfo`/PyMuPDF核实只有2页
   - 解决：查页数别只信系统提示，用 `pdfinfo` 或 PyMuPDF 核实
2. **Read工具打不开这份PDF**（报 `pdftoppm is not installed`）
   - 解决：写了 `pdf_to_images.py`（基于PyMuPDF，不依赖poppler）渲染成jpg再看，已进skill
3. **代数卷的"真插图占比"远低于几何卷**
   - 现象：这份卷子裁出83张小图，只有5张是真插图（第9题4张选择项图+第17题1张应用题图），其余全是学生手绘验算草稿（数轴、抛物线草稿）
   - 解决：不能假设"有候选图=一定要配图"，要打开看内容；这个预期和几何卷正好相反，已写进skill提醒
4. **`match_images_by_position.py` 的题号识别有两类失手**
   - 第7题的题号文字被背面透字冲花到完全无法识别，图片正确落进"未分配"区（没有被瞎猜给别的题）
   - 第25题"25."被OCR多粘一位数字读成"25.8"，触发反小数误判规则被跳过，体现为第24题候选数偏多——脚本已用"只找期望范围内最接近候选"的方式规避了更严重的连锁误判（一段被OCR读花的红笔批改文字偶然拼出"24."，一度挡住了19-23题的识别，这个bug在这次调试里被发现并修复）
5. **手写答案识别本身就不该做**（本次最重要的一条反思）
   - 现象：为了确认第6/9/10/11/13/14题的手写答案，做了多轮高分辨率裁图+2次MiniMax-M3交叉核对，这部分工作量占了这次任务的近一半
   - 原因：处理者（Claude）自己扩大了范围——用户最初要的只是"结构化题目"，没要求提取手写答案/批改结果，是看到"这个信息也能提取"就顺手做了
   - 解决：用户指出后，skill 改成默认五步流程不含手写答案识别，`answer` 字段默认省略，只有用户明确要求时才走"可选：提取手写答案"这一节
6. **手写答案有时候数学上就是错的**：第6题学生圈的是D，但数学上B（y=|x|）才是在(-∞,0)上单调递减的正确答案——如实记录卷面标注，不要自己"纠正"（这条原则仍保留在"可选"那一节里，只是不再是默认要做的事）

## 数据
- 裁剪图总数：83张；真插图：5张（约6%）；其余94%是手写噪声
- 位置匹配题号识别：19题中17个一次识别成功，2个（第7、25题）需要人工介入（背面透字冲花 / OCR多粘数字）
- 手写答案识别环节：约占当次总工具调用量的40%~50%（事后被判定为不必要的工作量）

## 操作记录
mineru_extract.py（PDF直接整份提交）→ match_images_by_position.py（位置匹配，识别出17/19题号，2题需人工核对）→ Claude读印刷题干整理JSON → MiniMax-M3交叉核对手写答案（当时做了，现已改为默认不做）→ latexify_questions.py 转LaTeX（scratchpad模式操作，云盘同步安全）→ render_markdown.py

## 一句话总结
代数卷候选插图大多是学生手绘草稿不是真插图；PDF页数别信系统提示；手写答案默认不该识别——这条是事后被用户纠正的，也是这次任务里影响范围最大的一条经验。
