## 概况
- 来源: 单份扫描 PDF (`202510胜利初一数学.pdf`, 5.9 MB), 4 页 A4 纵向 (PyMuPDF 核实就是 4 页, 系统提示也准)
- 内容: 胜利中学初一年级数学学习成果展示, 25 题 (10 选择 + 5 填空 + 10 解答), 题型含柱体/正方体展开图/数轴/有理数/几何体三视图/有理数运算/直角三角形旋转/表格应用题/拆项法/数轴动点, 跨度大 (几何 + 代数 + 应用)
- 同一目录下已有两份之前处理的 `.structured` (NESTED 在子目录里 28 张图, SIBLING 在 root 旁 23 张图), 这次按 skill 流程重做并把产物放到 `202510胜利初一数学.structured/` (root 旁, 跟 SKILL.md 规范一致)

## 关键发现 (问题 → 原因 → 解决)

### 2. OCR 把"7"读成"D"——题 1 的学生答案涂改被当成题号
- 现象: page_01 上 Q1 后面 "( 7 )" 看起来像题号 7, 实际上学生手写答案"D"被 OCR 错读成"7"
- 后果: match 脚本的题号识别正则 `\d{1,3}\.\、．` 不会匹配到"( 7 )" 里的 7 (因为它不在题号位置格式里), 所以没污染候选, 但用户/读者看 full.md 会困惑
- 处理: 写 questions.json 时保持 Q1 stem 不变 ("下列图形, 不是柱体的是(　)"), OCR 出的"( 7 )" 不进结构化产物

### 3. Q20 / Q23 的印刷体公式不是 image, 不应被 match 脚本扫到
- 这两份卷子里 Q20 (4 个小题) 和 Q23 (拆项法步骤) 都是**印刷体 inline 公式**, OCR 当作 equation 而不是 image, 所以根本没裁剪出来
- 这跟胜利初四不同 (那里 Q23 是手写验算被当成图) - 印刷公式 vs 手写验算的图识别逻辑要分开看
- 处理: Q20 / Q23 都 image: null, 公式内容直接手写 LaTeX 进 stem

### 4. Q22 的表格是 HTML `<table>`, 不是 image
- OCR 用 `<table>` 标签包了 Q22 的表格, 所以没有作为图裁出
- 处理: questions.json stem 字段里用 markdown 表格 (`| 星期 | ... |`) 还原, 跟 SKILL.md 的 source.note 里"不数字化学生解题过程, 只保留印刷题干"一致

### 5. Q25 OCR 把右括号读成花括号
- 原文: $|a-8|+|b+2|=0$
- OCR: `|a-8}+|b+2|=0` (右 `|` 变 `}`)
- 处理: 手写时直接写 `|a-8|+|b+2|=0`, render 时 LaTeX 用 `|...|` 配对

### 6. WPSDrive 同步坑: 写入后旧文件被"复活"
- 第一次原子写完 (scratchpad → 云盘) 后, 验证发现 images/ 里有 35 张图 (我新写的 26 + 旧版残留 9 张: q08_3d / q11 / q12_a / q12_b / q18 / q20_2 / q20_4 / q22 / q23)
- 原因: WPSDrive 同步客户端的延迟让 rmtree 之后, 云端的旧版本又被拉回本地
- 处理: 重做, 用 `shutil.rmtree(images/)` 整目录删 (而非逐文件), sleep 2s 等同步反应过来, 再 `mkdir` + 原子写, 验证只剩 26 张

### 7. 同目录有之前处理的 `.structured` 副本, 路径不规范
- 同目录有 `202510胜利初一数学/structured/` (NESTED, 28 张图) 和 `202510胜利初一数学.structured/` (SIBLING, 23 张图), 两份之前的手工处理
- SKILL.md 规范要求产物放在"试卷所在文件夹下的 structured/", 源 PDF 在 root, 所以规范位置是 root 旁的 `.structured/` (SIBLING 位置)
- NESTED 不符合 skill 规范, 但保留备份方便回滚
- 处理: 旧两份都 backup 到 `_archive/old_runs_2026-07-05_pre_skill_rerun/`, 产物落到 root 旁的 `.structured/`

## 操作记录
mineru_extract.py (PDF 4 页 1 次提交, batch_id 9f58fe95) → match_images_by_position.py (25 个题号识别到, 但警告很多) → PDF 渲染 4 张原图到 scratchpad + 拼 42 张裁剪图联系表 → 视觉读 4 页 PDF + 42 张联系表 → 修正 match 脚本的漏配/误配 (11 处) → 手写 questions.json (25 题, 12 题有图, 全部手写 LaTeX) → latexify_questions.py --validate-only (校验通过: 没有字面反斜杠+n, $ 全部配对) → render_markdown.py → 用 atomic_write 拷回 WPSDrive 202510胜利初一数学.structured/ (首次 35 张图, 旧版复活 9 张; 重做 rmtree + 重新写, 剩 26 张) → 双重 SHA256 验证 questions.json 和 questions.md 稳定

## 一句话总结
match 脚本只能当扫描器, 真正配图必须自己对照 PDF 原图逐题确认 (这次 25 题里 11 题需要修正 match 输出); 初一试卷的"印刷体 inline 公式 (Q20/Q23) + HTML 表格 (Q22) + 选择题公式选项 (Q4 D)"都不会被 OCR 当成 image 裁出来, 这些内容都要手写进 stem 字段而不是塞 image 路径; WPSDrive 同步延迟会让"先 rmtree 再写新文件"的策略失效, 第一次必须整目录 rmtree 等 2s 再 mkdir 才能彻底替换.
