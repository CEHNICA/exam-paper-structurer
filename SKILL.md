# 试题结构化工具

用 MinerU（版面解析+OCR+插图裁剪）+ Claude 自己的视觉能力（读印刷题干、核对插图）+ MiniMax-M3（LaTeX 转换），把拍照/扫描的试卷变成 `questions.json` + `questions.md`，每道题配好对应插图。

**目的是记录题目本身，不是记录这份答卷的批改结果**。默认只提取印刷体的题干、选项、插图——手写的答案、批改、解题过程一律不识别、不放进 `questions.json`（`answer` 字段默认省略或留空）。这不是"偷懒少做一步"，是这个技能本来就该做的事：用户要的是题库，不是复刻一份带批改的答卷副本。只有当用户明确要求"把答案/学生做的结果也记下来"时，才按本文档末尾"可选：提取手写答案"那节的方法去做。

这个技能有明确的技术栈和固定流程（不是通用的"文档处理"技能），按下面的步骤走，不要自己另起炉灶设计新方案。

## 前置：找到工作目录和配置

脚本在 `scripts/`（`mineru_extract.py`、`match_images_by_position.py`、`make_contact_sheet.py`、`minimax_vision.py`、`latexify_questions.py`、`render_markdown.py`），凭证在 `scripts/config.json`（`mineru_token`、`minimax_api_key`、`minimax_base_url`、`minimax_text_model`、`minimax_vision_model`）。**如果这个文件不存在或者是第一次用这个 skill，先看 `SETUP.md` 走一遍环境搭建流程**（装依赖、申请 MinerU/MiniMax 凭证、生成 `config.json`），不要瞎编凭证或跳过这一步。

## 处理一份新试卷：六步流程

**开始前**：先看一眼 `references/history/INDEX.md`（一份卷子一行，几秒钟看完），了解过去处理别的卷子时踩过的坑，很可能提前避开同样的问题。然后用 TaskCreate 建好这六步对应的任务（包括第6步"记录处理历史"），全部标记完成才算这次任务结束——**不要在任务列表里漏掉第6步，也不要在它还没完成的时候就跟用户说"处理完了"**。这是目前保证"记录处理历史"这件事真的会被执行、不会被悄悄跳过的主要手段（skill 本身只是文档，没法强制执行，全靠这个习惯）。

### 1. 解析（MinerU）
```
python scripts/mineru_extract.py "<试卷照片/PDF所在文件夹>"
```
文件名的自然顺序（微信照片自带的时间戳）就是页面顺序，一般不用手动重排。这一步在 `<试卷文件夹>/mineru_output/<每页>/` 下生成 `full.md`、`content_list.json` 和裁剪出来的插图（`images/`）。

**如果试卷本身就是一份 PDF**（比如手机扫描/WeChat导出的"xxx.jpg.pdf"），直接把整个 PDF 传给 `mineru_extract.py` 就行（`is_ocr` 已经默认开启），不用先拆成一张张图片——MinerU 会把整份 PDF 的阅读顺序当成一份连续文档处理，这对第2步"按位置配图"反而更有利（不会有"跨照片文件边界"的问题）。

**查看 PDF 里到底有几页、每页内容是什么，直接读原图确认，不要只看系统提示的页数**：这次实测中，环境提示这份 PDF 有"20页"，但用 `pdfinfo`/PyMuPDF 核实后其实只有2页（每页是一张两页合影照片）——系统给的页数估计不总是准的，尤其是这种由 QuarkXPress 之类排版软件生成的 PDF。如果 Read 工具报 `pdftoppm is not installed` 没法直接看 PDF 页面，用：
```
python scripts/pdf_to_images.py <PDF路径> <输出文件夹>
```
把每页渲染成 jpg 再用 Read 工具看（这个脚本用 PyMuPDF，不依赖 poppler）。

### 2. 按位置把插图配到题号（主要手段，不是靠肉眼比形状）

```
python scripts/match_images_by_position.py <content_list.json路径1> [<content_list.json路径2> ...]
```

**这是给每道题分配插图的主要方法**，把 MinerU 每一页 `mineru_output/<某页>/*_content_list.json` 按试卷页面顺序全部传进去（多个文件会当成一份连续文档拼接，即使某题插图因排版被挤到"下一张照片"，只要传入顺序对，也能跨文件配上）。

原理很朴素：`content_list.json` 是按阅读顺序排列的列表，一道题的插图在物理排版上几乎总是紧跟在这道题题干之后、下一题题干之前。脚本会顺着"接下来该是第几题"往后找题号文字，把两个题号之间出现的所有 image 条目作为这道题的候选插图——这比"拼一整页的联系表、凭图形形状肉眼比对"要可靠得多：形状相似的两道题（比如两道都是"四边形+对角线交点O"的图）光靠眼睛比很容易认错，位置信息不需要比形状，只需要看这张图夹在哪两个题号之间。

脚本输出对每道题的候选插图列表，并且**主动标出以下几种情况，提醒你（Claude）自己去看一眼再定，不会替你悄悄猜**——这几种情况不是"甩给人工"的意思，是你拿自己的眼睛就能立刻解决的事，别把它们当成待办事项晾着：
- 某道题一个候选都没有——可能是这道题真的没配图（比如纯文字的"多边形内角和"这类题），也可能是候选图被并进了别的题窗口，打开原图和相邻题的候选看一眼就能判断出来。
- 某道题候选数偏多（≥3）——通常是因为紧挨着的下一道题的题号没被识别出来（比如题号文字被纸张背面透字冲花、或者被OCR粘连在上一题最后一个选项的同一行末尾），两道题的插图被合并进了一个窗口。把候选图挨个读一眼，分清楚哪些真的是插图、哪些是红笔解题过程或背面透字的"鬼影"噪声，该给下一题的直接分过去。
- 有"未分配"的图片——出现在第一个题号标记之前，说明 MinerU 的版面阅读顺序本身也不完全可靠（这次实测里，第7题的图就因为版面判断问题被排到了整份文档最前面，比标题都早）。这类图不能瞎猜属于哪题，但也不是解决不了——结合题干内容（比如"小明沿六边形路径走"对应的就该是那张六边形图）你自己一眼就能对上。

**候选数正好是1、又没有警告的题，基本可以直接采用，不需要再额外核对形状**——这正是这个方法比肉眼比对更值得信任的地方。候选数≠1或者出现警告的题，就直接打开那一两张候选图看内容确定（候选集通常只有一两张，直接读图就行，不必再上"联系表"）——这一步该你自己动手做，不要因为脚本打了个"警告"标签就误以为这是需要停下来交给别人处理的事。如果一整页大部分题目都候选异常，或者想整体扫一眼某一页有哪些裁剪图，`make_contact_sheet.py` 仍然可以用来生成联系表辅助排查，但不再是默认第一步。

**不同科目/题型的"真插图占比"差别很大，别用一份卷子的经验套所有卷子**：几何题（陈毅初三数学那份）几乎每题都配图，候选图里大部分是真插图。代数/函数题（凤城高一数学那份）反而相反——大部分候选图其实是学生解题时随手画的草图（数轴、抛物线草稿、验算过程），真正印刷的插图很少，往往只有"下列图象中"这种明确要求看图判断的题，或者应用题里给的函数关系图。打开候选图看一眼就知道是印刷插图还是手绘草稿（草稿通常线条潦草、带箭头指向、和题干描述对不上），不要假设"有候选图=一定要配图"。

### 3. Claude 亲自读题（这一步没法完全脚本化，需要你自己判断）
用你自己的视觉能力直接读原始照片（不要只信 MinerU 的 OCR 文本——它对印刷体还行，但纸张背面透字、页面污渍这些噪声偶尔会让 OCR 读错或读丢印刷字符），完成：

- 把每道题**印刷体**的题干、选项转写成干净的文字。手写的答案标注、批改红笔、学生的解题过程一律不用管、不用识别——那是这份答卷的批改结果，不是题目本身，不属于这个技能要交付的东西（除非用户明确说要，见文末"可选"一节）。
- 结合上一步的位置匹配结果，把每道题最终确定的插图复制改名到 `<试卷文件夹>/structured/images/`，命名成 `q01.jpg`、`q02.jpg`……（一题多图就 `q25_fig1.jpg`/`q25_fig2.jpg` 这样）。
- 按 `references/questions_schema.md` 里的 schema 手写 `questions.json`（`answer` 字段默认省略，不用花时间去认手写答案）。

### 4. 转 LaTeX（MiniMax-M3 文本模式）
```
python scripts/latexify_questions.py <questions.json路径>
```
把题干/选项里的数学记号（∠、△、▱、√、分数、度数、平行垂直符号等）转成行内 LaTeX `$...$`。这一步会原地更新 `questions.json`（自动备份成 `.json.bak`）。

**这一步跑完脚本自己会打印校验结果，看它说了什么，不要假设"跑完就是对的"**：MiniMax-M3 偶尔会把 JSON 里的换行符双重转义，写成字面的反斜杠+n 两个字符而不是真正的换行。脚本内置的修复函数 `fix_and_validate()` 会自动处理并在结尾打印"校验通过"或列出剩余问题——**但修复规则本身要小心**：不能见到"反斜杠+n"就无脑替换成真换行，那样会连带破坏 `\neq`、`\notin`、`\nabla` 这些本来就以 `\n` 开头的合法 LaTeX 命令（这个坑真的发生过：`\neq` 被切成反斜杠+真换行+"eq"）。现在的规则是"n 后面不是小写字母才当真换行处理"（`\neq` 的 n 后面是字母 e，会被保留；真换行伪影后面通常是中文/"("/字符串结尾）。如果脚本打印了警告，别忽略，去看一眼是不是又出现了新的边界情况。

**模型名字容易踩坑**：这个账号是中国大陆 MiniMax Token Plan，base url 必须是 `https://api.minimaxi.com/v1/text/chatcompletion_v2`（不是国际站 `api.minimax.io`，那个会直接报 `invalid api key`，跟 key 本身有效性无关）。视觉模型用 `MiniMax-M3`——`MiniMax-VL-01` 在这个账号下不存在，`MiniMax-M1` 明确拒绝图片输入，`MiniMax-M2`/`MiniMax-M2.5`/`abab6.5s-chat` 会接受请求但实际上根本没看到图（答非所问）。`minimax_vision.py` 已经把这些坑踩过了，直接用就行，不要自己重新试模型名。

### 5. 生成最终 Markdown
```
python scripts/render_markdown.py <questions.json路径>
```
产出 `questions.md`，图片自动嵌入，选项用全角空格分隔，多问的 (1)(2) 会各自换行显示。

### 6. 记录处理历史（不是可做可不做的收尾动作，是流程的一部分）
在 `references/history/INDEX.md` 加一行，再写一份 `references/history/<卷子名>.md`（格式：概况/关键发现/数据/操作记录/一句话总结，参照已有的两份卷子）。不管这次处理得顺利还是踩了坑，只要有能让下一次处理更好的信息就记下来——这是这个 skill 能持续改进的唯一机制。这一步做完之前，这次任务在 TaskList 里不算完成，也不要跟用户说"都处理好了"。

### 产出位置
最终产物固定放在**试卷所在文件夹**下的 `structured/`（`questions.json` + `questions.md` + `images/`），和源照片放一起——不要放进这个 skill 目录或 Downloads 工具目录，那两个地方只是脚本/工作区，不是交付物的家。

## 云盘同步目录：一个真实踩过的坑

如果试卷文件夹在云同步盘里（路径包含 `WPSDrive`、`OneDrive`、`iCloudDrive`、`Google Drive` 之类的字样），**不要在那个目录里直接反复"读取→修改→写回→再读取"**。同步客户端会和本地写入抢文件，导致写入被静默回滚或损坏——这不是理论风险，是这次任务里真实复现过的问题（同一个文件连续两次读取，内容会不一致）。

正确做法：把要迭代修改的文件先复制到 Claude Code 的 scratchpad/临时目录（不会被任何云同步盘覆盖），所有中间步骤都在那边做，用延迟几秒后再读一次的方式确认内容稳定不变，最后再把定稿的文件一次性拷回云同步目录，拷贝后再验证一遍。第4步（LaTeX转换+校验）和任何需要多轮修改 JSON 的场合，都按这个模式来。

## 可选：提取手写答案

默认不做这一步——只有用户明确说"把答案/学生做的结果也记下来"之类的话，才执行下面的内容。这一步比听起来贵得多：手写字母连笔、多选题、涂改覆盖都容易认错，为了确认清楚经常要反复放大裁图、和另一个模型交叉核对，会显著拉长处理时间，所以不该是默认行为。

需要做时：
```
python scripts/minimax_vision.py <原始照片路径> "<问题：比如列出1-10题手写圈出的答案字母>"
```
用 MiniMax-M3 和 Claude 自己读图的结果交叉核对一遍。如果两次结果对不上，或者答案marker本身写得潦草（连笔字、覆盖修改、和邻近选项字母长得像），用 PIL 把那一小块区域单独裁出来放大再看一次，比反复通读整页更有效。

**如实记录标注的答案，不要自己"纠正"**：有时候手写圈的答案在数学上其实是错的（比如凤城高一数学卷子第6题，学生圈的是D，但正确答案应该是B，y=|x|才是在(-∞,0)上单调递减的函数）——这是学生的真实答卷，任务是准确转写卷面上标了什么，不是重新判卷。发现数学上对不上时，在 `source.note` 里如实说明"卷面标注是X，但数学上应为Y"就行，不要擅自把 `answer` 字段改成自己认为对的答案。
